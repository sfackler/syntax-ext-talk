Syntax Extensions

What, Why, and How

Steven Fackler / sfackler

What?

Macros! 


let count = try!(r.read(buf));
// Expands to...
let count = match r.read_buf(buf) {
    Ok(ok) => ok,
    Err(err) => return Err(err)
};

let foo = bytes!("hello");
// Expands to...
let foo = &[104u8, 101u8, 108u8, 108u8, 111u8];

Lexical

C's preprocessor

Very simple, operates on tokens - think sed 


#define MIN(a, b) ((a) < (b) ? (a) : (b))
#define struct union
#define DO_A_THING(t) do { \
        do_blah(t); \
        do_foo(t); \
    } while(0)

Syntactic

Lisps and Rust!

Operates on the abstract syntax tree (AST) 


macro_rules! try(
    ($e:expr) => (
        match $e {
            Ok(e) => e,
            Err(e) => return Err(e),
        }
    )
);

Why?

Macros can do things that normal code simply can't.

  • Return from the invoking function
  • (Almost) freeform syntax
  • Do work at compile time instead of run time!

Why not macro_rules?

It's not powerful enough!

Syntax extensions can run arbitrary code at compile time.

Take println! for example: 


test.rs:2:35: 2:37 error: argument never used
test.rs:2     println!("hello {}", "world", 10);
                                            ^~

How?

Syntax extensions don't have to be defined in the compiler anymore! (#11151)

They can be implemented in libraries that the compiler loads at runtime.

Example!

Let's write a syntax extension that creates sorted arrays: 


let foo = sort!("z", "hello", "a", "");
// Expands to...
let foo = ["", "a", "hello", "z"];

Caveats

  • Syntax extension libraries depend on libsyntax - avoid creating a runtime dependency on them!
  • Working with the AST can be extraordinarily verbose - try creating an impl block via AST nodes (or don't).
  • No type information.
  • Namespacing can be tricky. There's no way of knowing what path a certain item is accessible by.

⚠ Warning! ⚠

External syntax extensions are feature gated for a reason! The API can and will change.
